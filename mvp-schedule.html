MVP build schedule — Phase 0 + Phase 1

Date: 2026-06-04 · Team: 4 (Founder/PM + Backend + Mobile + Designer/4th eng) · Stack: React Native + Expo (merchant app), Python / FastAPI backend (as built; Node remains an option) · Cloud: AWS Cape Town · Endpoint: merchant-acceptance MVP feature-complete on pawaPay sandbox, code-frozen, ready for production cutover · Spec basis: 05-product-spec/

Every week's deliverables tie back to concrete items in the product spec (merchant onboarding + accept screens, the core Merchant / Transaction / LedgerEntry model with a double-entry ledger, and the REST + USSD channels in api-contracts.md). 19 weeks (~4.5 months) from kickoff to a feature-complete, code-frozen MVP on the pawaPay sandbox. The endpoint is "ready for production cutover" — Phase 2 (production deploy, closed beta, App Store + Play Store launch) is intentionally out of scope here and handled separately. The schedule front-loads slow external dependencies (legal counsel, pawaPay production KYC) so they're complete well before they could block Phase 2.

Wiggle-room philosophy

Schedules look fastest on paper and slowest in reality. This plan bakes in ~19% slack vs. the aggressive baseline (16 → 19 weeks) by:

  • Adding one explicit catch-up week at the end of Phase 0 (W5) — for anything that didn't quite land in W1–4 plus the first hello-world transaction.
  • Adding one explicit mid-phase catch-up week (W10, right after end-to-end round-trip works) — for whatever the first half of Phase 1 revealed.
  • Right-sizing per-week deliverables — every week has a single clear primary outcome, not a list of 5 things that all have to land Friday.
  • Calling out "slack to use" explicitly in catch-up weeks — they're not buffer to absorb later, they're work weeks with the team's own backlog as input.
  • Under-promising on milestones — the mid-MVP round-trip lands in W9, not W6, and code freeze is W19, not W13.

Scope locked at start

  • In: a merchant app (onboarding + accept flow); merchant onboarding (settlement account + till / short code; KYC flagged); customer-initiated USSD payment (scan the merchant's QR — a tel: USSD dial-through — or dial the till) with a charge-by-customer-number fallback; pawaPay-backed collect→bridge→settle; M-Pesa + Airtel + Orange (3 of 4 DRC operators); CDF + USD; French primary UI; MSISDN-based addressing; merchant auth.
  • Out: Afrimoney, bank-account settlement, rewards, bill pay, a separate consumer app, cards, direct operator integrations, Lingala/Swahili.
  • Out of THIS schedule (handled by Phase 2): production cutover, closed merchant beta, App Store + Play Store submission, public launch.

Week 7
Customer collection + USSD channel
Phase 1

F
Watch sandbox tests live with B; document every error path the team finds. Confirm pawaPay KYC completion status (likely approved by end of week).
B
Implement start_merchant_payment + payment_codes (generate the per-merchant tel: USSD code / till routing) and the USSD channel (channels/ussd/) plus the merchant-facing POST /transactions (create + collection trigger) per api-contracts.md. Watch: the PIN prompt happens on the customer's own phone via the operator, not in the merchant app — design the merchant's waiting UX to manage that expectation explicitly.
Week 8
Merchant settlement + charge-by-number fallback
Phase 1
Goal: the settlement leg (pass-through to the merchant) works end-to-end against sandbox, and the merchant-initiated fallback works; the full round-trip is one step away.
F
Test the settlement flow yourself with sandbox test numbers; document UX rough edges. Begin compiling FAQ / merchant-support runbook.
B
Settlement: deduct from the clearing account, call pawaPay Disbursement API to pay the merchant's settlement MSISDN (net of MDR), await callback, post final ledger entries (fee → revenue:fees; or refund the customer on failure). Failure-handling rules: when to retry vs refund vs require manual intervention.
M
Charge-by-customer-number fallback screen (merchant keys the customer's number to push a collection prompt to that customer's phone). Operator-inferred badge ("Charging an Airtel Money customer"). Request → pending → settled screens; net-of-fee shown to the merchant.
D
Merchant receipt / detail final implementation. Edge-case copy: customer number invalid, off-net failure messages, customer-refund messages.
Deliverable: 10 successful sandbox settlements to merchant test wallets across all 3 operators; merchant receipts generated and shareable; charge-by-number fallback verified.
Watch: a successful collection followed by a failed settlement must refund the customer — this path is easy to forget and confusing if it breaks.
Week 9
End-to-end round-trip
Phase 1
Goal: first real customer-to-merchant cross-network round-trip works in sandbox (collect from the customer → bridge → settle to the merchant). This is the technical mid-MVP milestone.
F
Run end-to-end demos with 5–10 stakeholders (advisors, pilot merchants). Capture confusion points. Update sandbox scenario doc.
B
Orchestrate the two-leg flow (customer collection → merchant settlement) as a single transaction with sub-states. Telemetry on per-leg latency. Failure pathway: collection succeeds but settlement fails → automatic refund to the customer.
M
The "request payment" CTA on the merchant home triggers the full flow; the customer pays the exact sticker price via USSD / QR. Progress indicator showing 2 legs. Polish: pull-to-refresh, optimistic UI, haptics on success.
D
End-to-end flow walkthrough video for the team. Iterate on confusion points found in demos. Polish merchant receipt screen.
Deliverable: 15 customer-to-merchant round-trips covering the 9 operator-pair combinations complete with terminal state in < 2 minutes each.
Watch: celebrate, but don't ship yet — every demo from here will reveal real-world edge cases. The next week is explicitly for absorbing them.
Mid-MVP milestone — end-to-end sandbox round-trip working.

Week 12
Reconciliation & observability
Phase 1
Goal: when something goes wrong, the team can see it. When the ledger drifts, a job catches it.
F
Design merchant-support runbook v1: top-10 likely issues and resolution steps. Choose support tooling (WhatsApp Business API or basic Intercom-style).
B
Daily reconciliation job: compare our ledger against pawaPay's transaction feed; flag mismatches and any settlement that didn't reach the merchant. Structured logging shipped to a queryable store. Sentry for errors.
M
Client-side error tracking (Sentry Native). Funnel telemetry (Mixpanel or Amplitude or self-rolled PostHog): merchant signup → onboarded → first-accepted-payment → repeat acceptance.
D
Status page mockups (for the case where one operator is down). User-facing error copy library — never blame the user.
Deliverable: Inject a fake mismatch into the ledger and see the reconciliation job alert. View live transaction stream in observability dashboard.
Watch: webhook callbacks WILL get dropped in production — the reconciliation job is the safety net, not an afterthought.

Week 14
Edge cases & failure modes
Phase 1
Goal: the long tail of "weird things that happen in production" is enumerated and handled.
F
Drive a "chaos demo" session — every team member tries to break the app for 2 hours. Bug-bash report. Settle: refund policy, dispute policy.
B
Handle: collection succeeds + settlement reversed by operator (refund the customer); collection reversed after settlement to the merchant (we eat the cost — alert + manual intervention); double-spend protection on idempotency; expired-OTP recovery.
M
Offline mode: queue write actions, show offline banner. Recovery flow when network returns. Background app state (don't lose in-flight transaction). Force-update flag in case we need to push an emergency fix.
D
"What went wrong" screens for every conceivable failure: no signal, wrong PIN, timeout, blocked card, insufficient float on the system side.
Deliverable: Bug-bash backlog cleared down to "polish only" (no correctness bugs). Edge-case matrix documented.
Watch: the "we eat the cost" branch is where the ledger gets confused fastest — model it carefully.

What's next: Phase 2

This schedule ends at the natural code-freeze point. Phase 2 (production cutover → closed beta → store submission → public beta) is a 4–6 week effort handled in its own plan, picking up wherever the Phase 1 team and pawaPay production-readiness happen to converge. See the roadmap Phase 2 section for the high-level shape.

The advantage of stopping at code freeze: Phase 2 is gate-able. If pawaPay production isn't quite ready, if counsel needs another week, if the team needs a breather — Phase 2 can start when conditions are favourable, not when the calendar says so.
